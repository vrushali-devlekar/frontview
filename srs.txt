It enables developers to connect GitHub repositories, trigger deployments, manage environment variables, stream live build logs, and roll back to previous versions — all from a single, clean dashboard.</p><p>This document defines the complete Software Requirements Specification (SRS) for DeployPilot at two levels:</p><ul><li>MVP Level — a hackathon-ready, fully functional prototype with core features demonstrating real-world utility.</li><li>Production Level — an enterprise-grade system with horizontal scaling, caching strategies, high/low-level design, and full system architecture.</li></ul><h2><strong>1.2 Problem Statement</strong></h2><p>Deploying a web application requires navigating multiple fragmented tools: a VPS, SSH access, process managers, reverse proxies, secret managers, and manual rollback scripts. DeployPilot unifies these concerns under a single, opinionated platform — removing DevOps overhead so developers can focus on code.</p><h2><strong>1.3 Objectives</strong></h2><ol><li>Provide a one-click GitHub OAuth login and repository connection flow.</li><li>Execute real deployments of Node.js/React applications via child processes or Docker containers.</li><li>Stream real-time build and runtime logs to the browser via WebSockets.</li><li>Securely store and inject environment variables using AES-256 encryption.</li><li>Enable versioned rollback to any previous deployment snapshot.</li><li>Expose an admin panel for user and deployment management.</li><li>Satisfy all hackathon mandatory requirements: Auth, DB, Admin, State Mgmt, Responsiveness, Gen AI, Deployment.</li><li>Meet production-level requirements: Redis caching, horizontal scaling, rate limiting, CI/CD pipeline, Docker orchestration.</li></ol><h2><strong>1.4 Scope</strong></h2><table><thead><tr><th><p><strong>Aspect</strong></p></th><th><p><strong>In Scope</strong></p></th><th><p><strong>Out of Scope</strong></p></th></tr><tr><th><p>Language Support</p></th><th><p>Node.js, React (static)</p></th><th><p>Python, Go, Java, Ruby runtimes</p></th></tr><tr><th><p>Auth</p></th><th><p>GitHub OAuth 2.0, JWT sessions</p></th><th><p>Google, GitLab OAuth (production backlog)</p></th></tr><tr><th><p>Deployments</p></th><th><p>Process-based (MVP), Docker (production)</p></th><th><p>Kubernetes cluster management</p></th></tr><tr><th><p>Database</p></th><th><p>MongoDB (primary)</p></th><th><p>Multi-region replication (future)</p></th></tr><tr><th><p>AI Integration</p></th><th><p>AI-assisted deployment suggestions &amp; Functional Requirements</strong></h1><h2><strong>4.1 Authentication &amp; Authorization</strong></h2><h3><strong>4.1.1 GitHub OAuth 2.0 Flow</strong></h3><ol><li>User clicks 'Login with GitHub' → redirected to GitHub authorization URL with client_id, scope=repo,user.</li><li>GitHub redirects back to /auth/github/callback with code.</li><li>Server exchanges code for access_token via POST to https://github.com/login/oauth/access_token.</li><li>Server fetches GitHub user profile (id, login, email, avatar_url).</li><li>Server upserts user document in MongoDB (create if first login, update token if existing).</li><li>Server signs a JWT (HS256, 7-day expiry) and sets it as an HttpOnly cookie.</li><li>User is redirected to /dashboard.</li></ol><h3><strong>4.1.2 Session Management</strong></h3><ul><li>JWT stored as HttpOnly, SameSite=Strict, Secure cookie — never in localStorage.</li><li>Every protected API route validates JWT via authMiddleware. Invalid/expired tokens return 401.</li><li>Logout endpoint clears cookie and (production) blacklists token in Redis with TTL matching remaining expiry.</li></ul><h3><strong>4.1.3 Role-Based Access</strong></h3><table><thead><tr><th><p><strong>developer</strong></p></th><th><p>Full CRUD on own projects/deployments/env vars. Read-only on others.</p></th></tr><tr><th><p><strong>admin</strong></p></th><th><p>All developer permissions + read/suspend all users + view platform-wide deployment data.</p></th></tr></thead></table><h2><strong>4.2 Repository Management</strong></h2><ul><li>GET /api/repos — calls GitHub API GET /user/repos (with per_page=100, sort=updated). Response cached in Redis for 5 minutes per user.</li><li>Client displays repo cards with: name, language, visibility (public/private), last updated, Stars.</li><li>Search filter (client-side) by repo name.</li><li>User selects a repo and branch to create a project.</li></ul><h2><strong>4.3 Project Management</strong></h2><ul><li>POST /api/projects — create project with { repoUrl, repoName, branch, projectName, userId }.</li><li>GET /api/projects — list all projects for authenticated user.</li><li>GET /api/projects/:id — project detail with all deployments.</li><li>DELETE /api/projects/:id — soft delete (isDeleted: true). Running deployments are stopped first.</li></ul><h2><strong>4.4 Deployment Engine</strong></h2><h3><strong>4.4.1 MVP — child_process Approach</strong></h3><ol><li>Receive deploy trigger for a projectId.</li><li>Create a Deployment document (status: queued) in MongoDB.</li><li>Clone repo: git clone --depth 1 --branch &lt;branch&gt; &lt;repoUrl&gt; /deployments/&lt;deploymentId&gt;.</li><li>Write .env file from decrypted environment variables into clone directory.</li><li>Run npm install via spawn. Pipe stdout/stderr to Socket.io room deploymentId:logs.</li><li>Run npm start (or detected start script) via spawn with --env NODE_ENV=production.</li><li>Assign an available port from the port pool (starting at 3100). Store in deployment.port.</li><li>Update Deployment status to running. Emit deployment:status event to client.</li></ol><table><thead><tr><th><p><strong>⚠ NOTE  </strong>MVP Constraint: Deployments share the host OS. Docker (P-01) provides proper isolation in production.</p></th></tr></thead></table><h3><strong>4.4.2 Production — Docker Approach</strong></h3><ol><li>Build Docker image: docker build -t deploypilot/&lt;deploymentId&gt; /deployments/&lt;deploymentId&gt;.</li><li>Run container with --network deploypilot --env-file .env --publish &lt;assignedPort&gt;:3000.</li><li>Store container ID in deployment document for lifecycle management.</li><li>Health check: poll GET http://localhost:&lt;port&gt;/health every 30s. Mark unhealthy after 3 failures.</li></ol><h2><strong>4.5 Real-Time Log Streaming</strong></h2><ul><li>Socket.io rooms: each deployment gets a dedicated room named dep:&lt;deploymentId&gt;.</li><li>Server: child process stdout/stderr → transform stream → emit log:line event with { timestamp, level, message }.</li><li>Client: auto-joins room on deployment detail page. Renders log lines in a styled terminal component (xterm.js or custom).</li><li>Log persistence: last 1000 lines stored in MongoDB deployment.logs array (MVP). Redis LIST with LPUSH + LTRIM 1000 in production for O(1) access.</li><li>Log levels auto-detected: lines containing ERROR/error → red, WARN/warn → yellow, default → white.</li></ul><h2><strong>4.6 AI Integration (Gen AI — Mandatory)</strong></h2><table><thead><tr><th><p><strong>✅ SUCCESS  </strong>Claude API (Anthropic) is used meaningfully — not as a gimmick. Two AI-powered features ship in MVP.</p></th></tr></thead></table><h3><strong>4.6.1 AI Log Diagnosis</strong></h3><ul><li>User can click 'Analyze with AI' on any failed deployment.</li><li>Last 200 log lines are sent to Claude API with system prompt: 'You are a senior DevOps engineer. Analyze this deployment log and: 1) identify the root cause, 2) provide a step-by-step fix, 3) flag any security issues.'</li><li>Response displayed in a structured panel: Root Cause / Fix Steps / Security Flags.</li></ul><h3><strong>4.6.2 AI Deployment Suggestions</strong></h3><ul><li>On project creation, Claude API analyzes the repo's package.json (fetched via GitHub API).</li><li>Returns: recommended start command, recommended Node.js version, environment variables the app likely needs (based on common patterns).</li><li>Suggestions pre-fill the deployment configuration form — user can override.</li></ul><h2><strong>4.7 Admin Panel</strong></h2><ul><li>Route: /admin — protected by isAdmin middleware.</li><li>User Management: paginated table of all users with columns: avatar, username, email, role, total projects, last active, status. The Redis Adapter solves this: every Socket.io emission is published to a Redis channel, all workers subscribe and re-emit to their local connected sockets. This makes WebSocket fan-out fully cluster-safe.</p><table><thead><tr><th><p><strong>Redis Channel</strong></p></th><th><p>socket.io#&lt;namespace&gt;#&lt;room&gt;#&lt;event&gt;</p></th></tr><tr><th><p><strong>Adapter package</strong></p></th><th><p>@socket.io/redis-adapter v8</p></th></tr><tr><th><p><strong>Redis connection</strong></p></th><th><p>Separate pub + sub clients required by Redis protocol</p></th></tr><tr><th><p><strong>Latency overhead</strong></p></th><th><p>~1–3ms additional per cross-instance event (negligible)</p></th></tr></thead></table><h2><strong>10.5 Deployment Queue (BullMQ)</strong></h2><p>In production, concurrent deployment requests are managed via a BullMQ job queue backed by Redis. This prevents resource exhaustion from simultaneous builds and enables retry logic, priority queuing, and rate limits per user.</p><ul><li>Queue name: deployments. After 3 failures: status = failed.</li><li>Bull Board UI mounted at /admin/queues for job monitoring.</li></ul><table><thead><tr><th><p><strong>SECTION 11 — DEPLOYMENT &amp; CI/CD PIPELINE</strong></p></th></tr></thead></table><h1><strong>11.
